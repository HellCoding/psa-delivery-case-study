# 08. 오프쇼어 산출물 비교 매트릭스

## 목적

두 오프쇼어 팀이 같은 PSA 요구사항을 구현했을 때, 최종 판단은 "더 많은 화면을 만든 팀"이나 "더 빠르게 끝낸 팀"으로 결정할 수 없습니다. 기능 완성도, QA 안정성, 코드 품질, 문서화, 커뮤니케이션, 후속 유지보수 가능성을 함께 봐야 합니다.

아래 표는 실제 벤더명과 내부 점수를 제거하고 공개용으로 재작성한 비교 구조입니다.

## 요약 비교

| 기준 | Vendor A | Vendor B | 판단 관점 |
|---|---|---|---|
| 팀 구성 | 소규모 단일 책임 구조 | 분업형 팀 구조 | 규모보다 실행 일관성과 책임 소재 확인 |
| 일정 | 안정화 중심의 긴 호흡 | 빠른 구현 중심의 짧은 호흡 | 속도와 완성도 균형 |
| 기능 완성도 | 핵심 흐름 대부분 안정 | 일부 UX 강점, 주요 결함 잔존 | PRD 필수 규칙 충족 여부 |
| QA 결과 | 치명 결함 재검증 후 안정 | Submit/Reject/Data sync 결함 잔존 | 업무 규칙 위반 여부 |
| 코드 품질 | Critical/High 0, Medium/Low 일부 남음 | Critical 0, High 일부 문서화, Medium/Low 대량 정리 | 운영 전 리스크 성격 |
| 문서화 | 회의/댓글 중심, formal artifact 부족 | PRD/WBS/기능명세/품질보고 풍부 | 인수인계 가능성 |
| 커뮤니케이션 | Jira 댓글과 수정 대응 강점 | 정기 보고와 문서화 강점 | PM이 통제 가능한 정보 흐름 |
| 최종 판단 | baseline 후보로 적합 | UX/문서화 일부를 후속 개선에 반영 | 안정성과 개선 아이디어 분리 |

## 가중 평가 예시

| 기준 | 가중치 | Vendor A | Vendor B | 검토 관점 |
|---|---:|---:|---:|---|
| 요구사항 충족도 | 30% | 4.5 | 3.5 | PRD 필수 업무 규칙 충족 |
| QA 안정성 | 25% | 4.5 | 3.0 | High 결함 잔존 여부와 재검증 |
| 코드 품질 | 15% | 4.0 | 3.5 | Critical/High/Medium/Low 처리 상태 |
| 문서화 | 10% | 2.5 | 4.5 | 인수인계와 추적성 |
| 일정/실행력 | 10% | 3.5 | 4.0 | 계획 대비 완료와 blocker 관리 |
| UX/확장 아이디어 | 10% | 3.5 | 4.0 | 대시보드, 리포트, 탐색 흐름 |

## 기능별 판단 예시

| 영역 | Vendor A | Vendor B | 권고 |
|---|---|---|---|
| Timesheet Submit | 제출 후 수정 제한까지 안정화 | 제출 후 버튼/셀 활성화 이슈 | Vendor A 기준 채택 |
| Reject Workflow | 반려 사유와 상태 전이 안정 | 반려 사유 입력 없이 Reject 가능 | Vendor A 기준 채택 |
| Project List | 목록 중복/관계 오류 없음 | join 조건 문제로 중복 row 발생 | Vendor A 기준 채택 |
| Report Builder | 실행 오류 수정 후 안정 | 기본 흐름은 구현, 검증 범위 제한 | Vendor A 기준 채택 |
| Dashboard UX | 기본 KPI 중심 | 카드/탐색/시각화 측면 강점 | Vendor B 아이디어 후속 반영 |
| Documentation | 최종 인수인계 보완 필요 | 문서 체계 우수 | Vendor B 방식 일부 채택 |

## 의사결정 입력값

- PRD 필수 요구사항 통과율
- Priority 1 UAT 통과/실패 요약
- High 이상 QA finding 잔존 여부
- ODC Code Quality gate 결과
- Jira 댓글과 재검증 기록의 충분성
- Confluence 문서의 최신성/인수인계성
- 후속 backlog로 안전하게 미룰 수 있는 항목과 미룰 수 없는 항목

## 공개 저장소 주의사항

실제 벤더명, 실제 점수, 계약 조건, 상업 정보는 공개 저장소에 포함하지 않습니다. 공개용 문서에서는 Vendor A/B 같은 일반화된 표현만 사용합니다.
