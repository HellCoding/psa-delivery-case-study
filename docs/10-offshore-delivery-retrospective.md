# 10. 오프쇼어 딜리버리 회고

## 핵심 교훈

같은 PRD를 전달해도 오프쇼어 팀의 결과물은 일정, 문서화, 구현 안정성, QA 대응 방식에 따라 크게 달라집니다. PM 관점에서 중요한 일은 단순히 업무를 배정하는 것이 아니라, 요구사항을 검증 가능한 단위로 나누고, 구현 결과를 증거 기반으로 비교하며, 어떤 산출물을 baseline으로 삼을지 설명 가능한 기준을 만드는 것입니다.

## 잘 작동한 방식

| 방식 | 효과 |
|---|---|
| PRD를 FR/BR 단위로 쪼갬 | Jira task와 QA scenario로 연결하기 쉬웠다 |
| Phase Epic 구조 사용 | 일정과 범위 변경을 팀별로 비교하기 쉬웠다 |
| QA 댓글에 재현 단계와 근거 요구사항 포함 | 개발자가 결함을 빠르게 이해하고 수정할 수 있었다 |
| Code Quality gate를 수치가 아닌 리스크 기준으로 해석 | PoC 수용 가능 항목과 운영 전 필수 조치를 분리했다 |
| 최종 비교표를 기능/품질/문서화/커뮤니케이션으로 분리 | 감정적 선호가 아니라 근거 기반 baseline 결정을 만들었다 |

## 어려웠던 지점

| 이슈 | 대응 |
|---|---|
| 팀별 Jira/Confluence 운영 방식 차이 | 결과물을 같은 평가 기준으로 다시 정렬 |
| 빠른 구현과 안정성의 trade-off | Priority 1 UAT와 High finding 잔존 여부를 우선 판단 |
| ODC 품질 점수 해석 차이 | Critical/High/Medium/Low finding의 운영 영향도를 별도로 문서화 |
| 화면상 동작과 서버 검증의 차이 | UI 테스트와 데이터/상태 동기화 확인을 함께 수행 |
| 문서 수량과 문서 품질의 차이 | 추적성과 인수인계 가능성을 기준으로 판단 |

## PM 관점의 개선점

- PRD 초기 버전부터 "수정 가능한 상태"와 "수정 불가능한 상태"를 표로 명확히 정의한다.
- WeeklyTimesheet, TimeEntry처럼 동기화가 필요한 entity는 상태 전이 다이어그램을 먼저 공유한다.
- Dynamic Report처럼 구현 자유도가 높은 기능은 allow-list, parameter binding, empty state를 초기 AC에 넣는다.
- 코드 품질 게이트는 프로젝트 시작 시점에 목표 점수보다 finding 처리 정책을 먼저 합의한다.
- Confluence 필수 문서 목록은 수량보다 업데이트 책임자와 갱신 주기를 함께 정의한다.

## 공개 포트폴리오로 재작성한 이유

실제 회사 자료, 내부 링크, 화면 캡처, Jira/Confluence 원문은 공개할 수 없습니다. 대신 이 저장소는 업무에서 사용한 판단 구조를 공개 가능한 형태로 다시 작성했습니다. 즉, 결과물 자체가 아니라 "요구사항을 구조화하고, 오프쇼어 산출물을 검증하며, 품질과 리스크를 근거로 의사결정하는 방식"을 보여주는 포트폴리오입니다.
